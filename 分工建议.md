# 小组分工建议

本项目为 Qt/C++ 本地可执行的个人发展规划系统，围绕“记录大学四年成长轨迹、分析当前能力画像、规划未来职业目标”展开。建议 5 名成员按模块分工，每个人负责一个相对独立的功能部分，同时在最终阶段进行集成测试与演示配合。

## 成员一：界面框架与交互集成

负责文件：

- `forms/mainwindow.ui`
- `include/mainwindow.h`
- `src/mainwindow.cpp`
- `curriculum vitae.pro`

主要工作：

- 搭建 Qt Widgets 主窗口，设计左侧导航栏和多页面结构。
- 使用 `QStackedWidget` 管理首页总览、课程与 GPA、项目/竞赛/实习、成就与角色、简历导出等页面。
- 负责按钮点击事件、页面切换、首页刷新逻辑和各功能模块的 UI 集成。
- 统一界面视觉风格，包括深色侧边栏、卡片式统计信息、按钮、输入框、下拉框、表格、进度条等控件样式。
- 处理首页内容较多的问题，将首页改为可滚动区域，保证雷达图和职业路线图都能完整展示。
- 修复交互细节问题，例如下拉框选项 hover 时文字不可见的问题。

## 成员二：数据模型与数据管理

负责文件：

- `include/models.h`
- `src/models.cpp`
- `include/datastore.h`
- `src/datastore.cpp`

主要工作：

- 设计课程记录、经历记录、成就记录和个人资料等核心数据结构。
- 为各类数据模型实现 JSON 序列化与反序列化，支持数据保存和读取。
- 实现 `DataStore` 数据管理类，统一管理个人资料、课程、经历、成就等数据。
- 实现 GPA 计算、累计学分计算、成长建议生成等基础业务逻辑。
- 设计并维护演示数据，使课堂展示时可以一键载入完整样例。
- 为创新功能提供数据支持，包括能力雷达评分和职业目标规则分析。

## 成员三：核心功能模块实现

负责内容：

- 课程记录模块
- GPA 计算模块
- 项目/竞赛/实习经历模块
- 成就与角色模块

主要工作：

- 实现课程添加、删除、表格展示功能，支持学期、课程名称、学分、成绩和绩点展示。
- 根据课程成绩和学分自动计算 GPA，首页同步展示 GPA 和累计学分。
- 实现项目、竞赛、实习、课外活动等经历的录入、删除和展示。
- 实现奖项、荣誉、角色、志愿服务等成就记录的录入、删除和展示。
- 完成必要的输入校验，例如课程名称不能为空、学分必须大于 0、经历名称不能为空、成就名称不能为空。
- 配合界面负责人完成数据录入后自动刷新首页统计信息、雷达图和路线图。

## 成员四：创新功能一，能力雷达图与成长画像分析

负责文件：

- `include/radarchartwidget.h`
- `src/radarchartwidget.cpp`
- `src/datastore.cpp` 中能力评分相关逻辑
- `src/mainwindow.cpp` 中首页雷达图展示逻辑

主要工作：

- 设计五维能力画像：学业基础、实践能力、竞赛科研、荣誉成果、领导协作。
- 根据课程、GPA、项目/竞赛/实习、成就和角色记录自动计算能力分数。
- 实现 `abilityScores()` 和 `abilitySummary()`，生成能力分数和文字解读。
- 使用 Qt 原生 `QPainter` 自绘雷达图，不依赖 QtCharts 或第三方库。
- 绘制雷达图网格、坐标轴、能力多边形、节点和标签，使能力画像直观可视化。
- 处理空数据状态，避免误导性分数，并提示用户先录入成长档案。

## 成员五：创新功能二，职业目标路线图与简历导出

负责文件：

- `src/datastore.cpp` 中职业目标分析相关逻辑
- `include/datastore.h` 中职业规划相关结构体定义
- `src/resumeexporter.cpp`
- `README.md`
- `docs/演示说明.md`

主要工作：

- 设计职业目标规则库，支持数据分析师、软件开发工程师、金融科技/金融分析、保研/科研方向、产品/项目管理方向。
- 根据能力雷达分数和目标关键词计算职业目标匹配度。
- 生成优势分析、短板分析和大一到大四的阶段性行动路线图。
- 在首页展示目标选择框、匹配度进度条、优势短板说明和四年行动路线表。
- 实现 Markdown 简历导出，包含个人信息、课程、经历、成就、能力雷达画像、职业目标规划摘要和成长建议。
- 维护项目说明文档，确保文档内容与最终程序功能一致。

## 协作说明

- `mainwindow.cpp` 和 `datastore.cpp` 是共享文件，实际开发中容易被多人同时修改，因此建议由界面负责人和数据负责人统一集成。
- 创新功能负责人可以负责算法和功能设计，再由界面负责人统一接入主窗口。
- 提交 Git 时，如果代码已经定稿，建议按最终文件归属提交，避免多人重复提交同一个文件。
- 汇报时建议前 3 名成员讲核心功能，后 2 名成员讲创新功能，这样结构清晰、亮点突出。
